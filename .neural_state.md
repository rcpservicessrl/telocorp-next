## ⚠️ LECCIONES APRENDIDAS (PARA OTRO AGENTE)

1. **Admin y Cliente están SEPARADOS:** El admin usa `/admin/login` (compuerta oculta), tiene su propio layout sin Header/Footer de cliente. El `ClientShell` component detecta `pathname.startsWith('/admin')` y no renderiza la shell del cliente.

2. **Deploy via Vercel CLI:** `cd c:\dev\telocorp-next && echo y| npx vercel --prod --yes`. Los archivos >50MB fallan por CLI — usar GitHub para archivos grandes.

3. **Imágenes de productos:** Las URLs en Supabase apuntan a `https://raw.githubusercontent.com/rcpservicessrl/telocorp-next/master/public/TeloCorp/images/imageX.png`. El componente de producto construye: `wsrv.nl/?url={URL}&w=400&output=webp`.

4. **Supabase Auth con SSR:** Se usa `@supabase/ssr` tanto en browser (`createBrowserClient`) como en server (`createServerClient`). El middleware refresca la sesión en cada request.

5. **Login redirect:** Después de signIn, se usa `window.location.href` (no `router.push`) para que el middleware detecte las cookies nuevas.

6. **StatusChanger genérico:** Un solo componente para cambiar estado inline en admin (funciona con cualquier tabla: repara_bookings, instala_bookings, lleva_requests, orders). Auto-notifica al cliente.

7. **Chat widget con fallback:** Si Gemini falla (429/503/cualquier error), responde con keywords inteligentes. No se queda colgado.

8. **Gemini API key (AQ.):** Las keys AQ. son el formato nuevo permanente. Requieren que "Generative Language API" esté HABILITADA en Google Cloud Console. Si la key fue creada antes de habilitar, regenerar.

9. **WhatsApp del negocio:** +1 (809) 903-8707 — usar en todo (chat fallback, footer, notificaciones).

10. **Node.js 26 en esta máquina:** SWC de Next.js no es compatible con la arquitectura, usa WASM fallback. Funciona pero es más lento en builds locales. En Vercel compila normalmente.
